Genesis · Design Spec

Session Launcher

An Omnara-inspired new-session experience: surface the sticky session bindings — workspace, engine, model, effort, and repo-root-vs-worktree — up front where they actually take effect, add a first-class New Workspace action (incl. clone-by-URL), and let the session list group by workspace / worktree / session. Hybrid: the chat-first quick-start stays; the configurator is the enriched landing.

Why

The problem this closes, and the insight from Omnara.

Genesis is chat-first: you start typing and the first turn binds workspace + engine + model, then those selectors lock (workspaceLocked = messages.length>0; interactive model spawn-pinned after the first assistant turn). The controls live in the composer — the one place they can't be freely changed once the session is real.

Omnara inverts this: a New Session configurator presents every binding before the session exists — Using host · start engine · model · effort · in workspace · on root|worktree → Start Session — plus New Workspace and a sidebar that groups by Workspaces / Worktrees / Sessions. The realization: Genesis already has almost all of these primitives — they're just scattered in the composer and exposed after they've locked. The launcher relocates them to create-time (when they bind) and adds the two genuinely-new bits: a per-session root/worktree toggle and session grouping.

The launcher (hybrid)

Quick-start stays for speed; the configurator is the enriched New Session landing, collapsible via show more / less.

  • Rows are the sticky bindings. Each maps to an existing server contract: engine → engine, model → model, effort → effort, workspace → workspaceId (all already parsed by parseChatRequest in chat-sdk.ts). The launcher writes them into the first /api/chat call — no new binding mechanics, just an earlier surface.
  • Quick-start unchanged: typing in an empty composer still works with defaults (last-used workspace/engine/model), so speed isn't taxed. The card is the option, not a gate.
  • Root vs worktree becomes a per-session choice that sets the run cwd (workspace.rootPath vs a cut worktree) — plumbs a new noWorktree-equivalent per turn-1, on top of BRO-1512's per-workspace default.

Sidebar grouping

The 2nd screenshot: a Workspaces / Worktrees / Sessions switch + Recent sort over the (86+) thread list.

Today the drawer is a flat newest-first list (BRO-1567). Add a grouping selector: Sessions (today's flat list), Workspaces (threads grouped under their bound workspace — the DTO already carries workspaceId/workspaceName via listThreads), Worktrees (grouped by the per-session worktree branch when not root-mode). Pure client-side projection over data the thread summary already returns — no engine change for Sessions/Workspaces; Worktrees needs the worktree label surfaced in the summary.

Phasing

Build order (decided): slice 5 → launcher card → grouping → host picker.

A

New Workspace by git URL — BRO-1629 slice 5 first

POST /workspaces {gitUrl} → validate URL (block file://, credentialed ssh, SSRF hosts) → clone into the GENESIS_PROJECTS_ROOT allow-root → register via FsWorkspaceRepository + idempotent registerWorkspace. PWA "New Workspace" affordance gains a URL field beside discover→pick. Security spine intact: client names a URL, never a path. Standalone; ships value immediately.

B

Launcher configurator card + root/worktree toggle

Relocate engine/model/effort/workspace selectors into a pre-session card (reusing the existing PromptInput* selects + body params); add the explicit Start Session and the per-session root/worktree toggle. Finish BRO-1622 engine capability-gating here. Keep quick-start.

C

Sidebar grouping / filter

Workspaces / Worktrees / Sessions switch + Recent sort. Mostly client-side over listThreads; surface the worktree label in the thread summary for the Worktrees mode.

D

Host picker deferred

Only when multi-host is real. The HostProvider seam already exists; a picker would select the lease target. No work until a 2nd host is deployed.

Decisions & open questions

Decided

  • Hybrid session-init — quick-start retained, configurator is the enriched landing.
  • Slice 5 (git-URL clone) builds first.
  • Host picker deferred — single host today.

Open

  • Does Start Session create an empty thread eagerly, or defer thread creation to the first message (keeps ghost-thread count down)?
  • Worktree mode: expose branch name at create, or auto-name genesis/session-<id>?
  • Grouping: server-side worktree label on the summary, or derive client-side from the session's branch?
Testability: BRO-1634 (composer not headlessly drivable) applies doubly to the launcher — build it with stable data-testids so the New-Session flow is E2E-checkable from day one.
