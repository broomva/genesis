Genesis · Phase 2.5 · Design Spec

The workspace registry should be a place, not a row

Killing the env-var / restart hatch by modelling a workspace as an FS-native, git-backed substrate — reached through a repository port, indexed by a disposable projection, reconciled by a controller. The same architecture Genesis already reaches for, turned on the registry itself.

BRO-1629 epic BRO-1356 / BRO-1357 follows BRO-1627 2026-07-01 draft · awaiting greenlight

Two complaints motivate this: the current workspace registry is admin-only (you edit an env var), restart-required (a boot snapshot), and can't be driven from the PWA (no write verb). This is a structural gap, not a config nit — and the fix reuses ~80% of the plumbing BRO-1627 already shipped.

01 Why the env approach is structurally stuck

Three facts from the code, and together they mean there is no runtime write path:

FactWhereConsequence
The registry is a boot snapshot. workspaceRegistry is private readonly, populated once in the constructor from env-derived config.supervisor.ts:171, 200–214Nothing mutates it after boot.
Persistence is one-directional. ensureWorkspace() only pushes the in-memory registry → DB.supervisor.ts:217–232The workspaces table is effectively write-only; a restart rebuilds purely from env, and a row added any other way is invisible.
The HTTP surface is read-only. GET /workspaces, no POST/DELETE.server.ts:176The PWA has no verb to call.
The good newsThe workspaces table (db/schema.ts:13), store.upsertWorkspace (store.ts:87), the ws-default id reservation, the sticky binding + the "workspace was deconfigured since the thread bound it" handling (which already anticipates workspaces coming and going), scanGitRepos (apps/api/src/workspaces.ts), and the picker UI all exist. What's missing is a mutable, DB-backed registry and two endpoints.

02 Thesis — a workspace is a place; the source of truth is versioned files

Genesis already defines its runtime with one load-bearing line from the greenfield spec:

Host-invariance insight (BRO-1356)Runtime = { filesystem + bash + egress }. Local, VPS, and microVM are interchangeable behind ExecutionHost.

So the filesystem isn't an implementation detail you build a workspace table on top of — it is the substrate the agent natively acts in. The AI-native move: model the registry, the metadata, the session state, and the control plane itself as files the agent reads and writes with its own tools. Git is a content-addressed database the agent can natively read, write, diff, and sync — which dissolves the DB-vs-FS tension: you get durable, stable identity and an FS the agent operates on directly. Manifest-in-git is a database (content-addressed, durable, stable-identity, syncable) that also happens to be the agent's native medium. You don't give up the DB properties — you pick a DB the agent can git log.

03 The layers (and the pattern each one is)

1

Substrate

file-based routing

A well-known root; a project is a directory. Discovery is readdir, not a query. The layout is the API (Next.js/Rails routing applied to workspaces).

2

Manifest

declarative desired-state

The metadata the FS can't hold lives in a sidecar the agent reads natively (.genesis/workspace.toml). The stable id lives in the file, so renaming/moving the dir never breaks a session's sticky binding (Kubernetes/IaC pattern).

3

Git as the transaction log

event sourcing

Every mutation (add/remove/rename/archive) is a commit → durability, history, audit, rollback, multi-machine sync for free; portable across ephemeral hosts (a microVM materializes a workspace by cloning the manifest's gitUrl).

4

Repository port

hexagonal

The domain talks to a WorkspaceRepository interface; the FS-adapter is one impl. Same seam instinct as ExecutionHost + Store already in the codebase — leaves the door open for a SQLite index later, never as truth.

5

Projection index

materialized view

A fast id→path / owner / phase lookup, built by scanning manifests + folding session NDJSON. Disposable: genesis reconcile rebuilds it. This is literally Genesis's own projection-reducer pattern (NDJSON → phase machine) applied to the registry.

6

Reconciliation controller

desired vs observed

A watch loop (inotify, or lazy-on-read) that materializes git-origin workspaces not yet checked out, marks vanished paths unavailable instead of dangling, and GCs stale worktrees (K8s-controller shape; slots into the Phase-5 observe→decide→act loop).

The stable id lives in the file, so renaming or moving the directory never breaks a session's sticky binding — that was the strongest DB argument, and a manifest field answers it without a database.

05 The repository port

The domain depends on an interface, not the filesystem — the same seam instinct as ExecutionHost and Store already in the codebase. The FS-adapter is one implementation; a SQLite index can be added later as a cache, never as truth.

interface WorkspaceRepository {
  list(owner?: string): Promise<WorkspaceView[]>;   // projection read (DTO: id,name,phase — no rootPath)
  resolve(id: string): Promise<Workspace | undefined>; // full record incl. rootPath (server-only)
  register(spec: RegisterSpec): Promise<WorkspaceView>; // discover→pick OR git-URL; server derives rootPath
  remove(id: string): Promise<void>;
  available(): Promise<ScannedRepo[]>;              // unregistered git repos under the allow-root
}

Supervisor.resolve() and listWorkspaces() route through this instead of the private Map. The current in-memory registry becomes the projection cache behind the FS-adapter — read-through-DB, not a per-process boot snapshot.

06 The security crux — the client must never name a path

Non-negotiablerootPath is an absolute server path. A POST /workspaces {rootPath} from the PWA is arbitrary-location code execution + ../ traversal. That is exactly why /workspaces and /health already strip rootPath (BRO-1627 P20). Filesystem authority never leaves the server.

A · Discover → pick 2.5a

Admin sets an allow-root (GENESIS_PROJECTS_ROOT). GET /workspaces/available scans it for unregistered git repos (scan logic already exists). The PWA lists them; POST /workspaces registers by selecting one — the server derives rootPath and validates the resolved path stays inside the allow-root.

B · Add-by-git-URL 2.5b

The real self-serve path. POST /workspaces {gitUrl} → server clones into the managed root, then registers. Users add any repo by URL, never touching a path. Lines up with the existing microVM GENESIS_GIT_URL model.

The security boundary is a mount, not a WHERE clause: the agent's authority is precisely which directories it can see — allow-root, worktree isolation, microVM bind-mounts. Least privilege is filesystem scoping, which is what "Runtime = filesystem + bash + egress" already implies.

09 The honest costs (not free)

  • Concurrent writes. Two agents mutating the registry → git gives you merge, but conflicts need a resolver or single-writer discipline (a lock file, or route registry mutations through one supervisor). Design it in, don't discover it.
  • Queries beyond readdir. "All of Carlos's archived workspaces idle >7d" is a manifest table-scan — this is why the projection index (layer 5) earns its keep. Be honest that you still built an index; you just made it disposable.
  • Scan latency at scale → watch-driven projection, not scan-per-request.
  • Secrets never touch the manifest. The repo already learned this (secrets.env, 0600, never in git). Manifests hold references to credentials, not credentials.
  • Shared truth vs local materialization. The git remote is the sync point (desired, shared); the checked-out FS is per-host (observed). Keep that split explicit or you'll expect a microVM to see a checkout that only exists on your laptop.
  • Multi-instance consistency. resolve() checks the in-memory Map; across multiple API instances a workspace added on one wouldn't be seen by another → the registry wants read-through-DB (or cache-invalidation), not a per-process snapshot. 2.5a's DB-backed registry already moves toward this.

Recommendation

Ship Phase 2.5a first — it removes both of your complaints (restart + can't-drive-from-PWA) in ~a day, reuses scanGitRepos / upsertWorkspace / the id-reservation verbatim, and lays the WorkspaceRepository seam so 2.5b–d are additive, not rewrites. The FS-native git-manifest substrate (2.5c) is the north star; 2.5a is a stepping stone toward it, not a detour — the DB registry it introduces demotes cleanly to the projection index later.

The one fork to decide for 2.5a's PWA affordance: discover→pick only, add-by-git-URL only, or both. Discover→pick is safest + smallest and needs zero new infra; git-URL is the true self-serve path but adds a clone/managed-root surface. My default: discover→pick in 2.5a, git-URL fast-follow in 2.5b.
